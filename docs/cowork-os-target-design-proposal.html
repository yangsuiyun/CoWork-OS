Design-First Architecture Proposal

CoWork OS 目标设计方案

脱离当前代码实现，从第一性原理出发，给出 CoWork OS 的「最优目标设计」：在保留现有领域语言与上下文清单的前提下，补齐一致性边界、领域事件契约、安全内核与外部 Agent 持久会话，并提供「现状 → 目标 → 代价」的可增量演进路线。

2 层稳定内核 / 用户态能力层
4+5核心域 / 支撑域分组
3 缺口一致性·事件契约·失败语义
现在/演进/暂缓变更三档分级

0. 本文档定位

本文是对 docs/domain-driven-design-concept-model.html（代码派生的现状建模报告）的升一层版本： 前者回答「现在的代码长什么样」，本文回答「最优设计应该长什么样，以及如何从现状走过去」。

方法论：本方案由多视角专家评审（实用 / 架构 / 创新 / 运维 / 批判）综合得出。 所有外部项目对标一律降权为「灵感参考」而非权威依据；所有新增对象均按 现在 / 演进 / 暂缓 三档分级，并附迁移代价，避免「对象动物园」与纸上架构。
与现状报告的边界：本文是设计目标，不是逐项重构清单。落地时以「核心域不变量优先固化、支撑域给接口契约」为原则增量推进。

2. 设计原则（First Principles）

#原则含义为何最优
P1内核稳定，能力易变状态、安全、审计沉淀在内核；编排/技能/通道/协议放用户态。同构于 OS kernel/userland、k8s control-plane/data-plane，隔离变化半径。
P2执行即事实，事件即真相所有入口最终汇聚为 Task，过程以领域事件追加记录。恢复、时间线、审计、回放统一在一条事件链上。
P3默认零权限，按边界授予Agent/Task 默认不持有任何危险能力，按限界上下文显式授予。最小权限 / 零信任；Steward Agent 自然成立。
P4单向依赖（DAG）上层依赖下层，禁止反向；跨上下文一律走事件或仓储接口。避免循环依赖、权限绕行、状态分叉。
P5边界即防腐层（ACL）外部世界通过明确 ACL 接入：MCP=工具/数据，ACP/A2A=外部 Agent。外部协议变化不污染核心域语义。
P6失败显式，绝不静默失败有类型、有补偿、有对账；日志记录但不替代状态语义。day-2 可运维，符合 Fail-Fast。
P7人类在关键回路现实副作用、记忆固化、技能发布须经确定性治理或人工审核。强 Agent 安全不能依赖模型自觉。

3. 目标分层架构

把「变化快的 Agent 能力」放用户态，把「状态 / 安全 / 审计」留在稳定内核——这是 §11.1 方向的形式化版本，并补齐了内核中缺失的事件存储与能力授权。

用户态能力层 易变 · 可扩展

承接需求多样性，通过事件 / 仓储接口与内核协作，绝不反向耦合进内核。

Agent Orchestration Skills Runtime Channel Gateway MCP Connectors (Tool ACL) ACP / A2A (Agent ACL) Everything Workbench Memory & Workflow Intelligence

稳定内核 状态 · 安全 · 审计

系统的不变量与事实源所在；对上只暴露窄接口（命令 + 事件）。

Task Runtime（执行事实源） Event Store / Timeline（事件真相） Permission & Safety（能力授权） Workspace Governance（治理边界） Session Runtime（运行态拥有者）

通用域 工程支撑

Profiles / Settings Updater Telemetry Native Helpers Secure Storage
与现状的关键增量：内核显式纳入 Event Store（当前为 TaskEvent 追加日志，已具雏形）与 Capability 授权（当前为执行时询问式 PermissionEngine）。二者均给出双态演进（见 §6 / §7）。

4. 限界上下文（收敛分组视图）

现状报告列出 16 个上下文，认知负荷偏高。本方案保留完整清单，但以「4 核心域 + 5 支撑域 + 通用域」分组导航，并明确每组的上游/下游与所有权。

分组限界上下文上游（依赖谁）下游（被谁依赖）
核心域任务执行 / 代理编排 / 托管代理 / 权限与安全Event Store、Workspace几乎所有支撑域
支撑域记忆与 Workflow Intelligence / Everything Workbench / 通道网关 / 技能运行时 / MCP·外部 Agent 互操作Task 事件、WorkspaceUI / 自动化
通用域Profile / 设置 / 更新 / 遥测 / 控制平面与设备 / 报告评估可靠性—全局横切
收敛建议：「Inbox/关系代理」「浏览器/电脑使用」可作为能力归入对应支撑域（前者→通道网关 + 记忆，后者→技能运行时 + 权限），而非独立顶层上下文，减少边界碎片。

5. 聚合与不变量

最优设计的价值不在「更多对象」，而在「更少、更硬的不变量 + 明确的聚合根写入口径」。下表只固化核心域聚合。

聚合根只能经聚合根修改的子对象关键不变量并发策略
TaskTaskEvent、SessionSnapshot、Result、Approval 归属Canonical Task：rawPrompt → userPrompt → prompt，技能只附加上下文不替换定义。事件追加幂等（seq 单调）；状态由事件投影。
WorkspaceWorkspacePermissions、本地策略镜像 .cowork/policyWorkspace Capability First：网络/Shell/写/删先过工作区能力。乐观锁（version）。
ApprovalRequest审批状态机Hard Deny 不可被 mode/fallback 覆盖；data_export 独立审批。单写：仅 PermissionEngine 创建。
AgentTeamRun / OrchestrationGraphGraphNode、childTasks远程节点回写以 remoteTaskId 收敛为节点 output/error。节点级幂等更新。
ExternalAgentSession 演进Lease、Queue、RemoteRun、HeartbeatRemote Agent Is Not A Tool：委派前过 endpoint/能力/授权/数据外发审查。租约持有者单写 + 心跳续租。
反伪抽象：§11.4 提议的 Hook/Session 系列细粒度对象，本方案只在有两个真实消费者时才引入；初期以聚合内字段 + 事件表达，避免过度建模。

6. 领域事件契约

跨上下文协作一律走领域事件，事件名与 payload schema 版本化，杜绝语义漂移。事件是内核对用户态唯一的「外发真相」。

TaskCreated→ TaskEventAppended*→ ApprovalRequested?→ ArtifactCreated?→ TaskCompleted / Failed
事件生产者（唯一）消费者契约要点
TaskCreatedTask RuntimeDaemon、Renderer、Mission Control必带来源 origin（manual/cron/hook/api/channel）+ correlationId。
TaskEventAppendedSession Runtime / ToolSchedulerTimeline、Trace、Memory、ManagedSession mirrorseq 单调、actor 明确、payload 版本号。
ApprovalRequested / ResolvedPermissionEngineRenderer、ChannelSession、Task Runtime风险动作暂停，决策回写为事件而非直接改状态。
ArtifactCreated工具执行 / 文件检测Workbench、Task feed以 taskId 归属 + sha256 幂等。
TaskCompletedTask RuntimeMemory、Dreaming、Heartbeat触发记忆巩固 / 自学习候选（仅候选）。
RemoteAgentDelegated / ACPTaskUpdated 演进RemoteAgentInvokerOrchestrationGraph、Mission Control远程状态规范化回写，必带 requestId 幂等。

7. 一致性与事务边界 现状最大缺口

现状报告未明确各上下文是强一致还是最终一致——这是设计被质疑时的薄弱点。本方案显式定义。

关系一致性模型失败处理 / 补偿
Task ↔ 其 TaskEvent强一致（同聚合，事务内）事件写失败 → 命令整体失败，不产生半状态。
Task ↔ Artifact最终一致（事件驱动登记）落盘成功但事件失败 → 后台对账扫描 orphan artifact 补登记。
Task ↔ Memory最终一致（TaskCompleted 触发）记忆固化失败可重放事件；Dreaming 仅候选，不静默写。
Task ↔ ManagedSession最终一致（事件镜像）ManagedSession 是 backing Task 的投影，丢失可由事件重建。
内核 ↔ 远程 Agent最终一致 + 租约心跳丢失 → stale recovery；重复投递由 requestId 幂等吸收。
原则：聚合内强一致、跨聚合事件最终一致、外部集成经 ACL + 幂等。任何「假设成功」的隐式路径都要替换为显式对账。

8. 安全内核（Capability + Hook）

当前态：询问式 PermissionEngine 现在

工具调用时调用 PermissionEngine 产出 allow/deny/ask；hard-deny、危险命令、task restriction 不可覆盖；data_export 独立审批。

保留并强化：把 hard-deny 规则与审批通道做规则级测试矩阵。

目标态：能力令牌 + 确定性 Hook 演进

Task/Agent 启动时被授予一束可吊销能力令牌（按 bounded context）；工具调用须持令牌；hard-deny 是「不可铸造的能力」。

PreToolUse / PostToolUse Hook 作为 PermissionEngine 的确定性前后置治理点（从「提示词规则」升级为可审计拦截）。

不变量（两态共同）：强 Agent 会产生现实副作用，安全绝不依赖模型自觉。发送 / 删除 / 部署 / 数据外发必须先治理再执行。

9. 外部 Agent 持久会话 高价值

外部 Agent 是「受治理的协作者」而非工具。借鉴 actor / 租约模型（参考 acpx warm session owner，作为灵感），把一次委派建模为持久会话而非 HTTP 调用。

acp_discover→ 能力声明 + 授权→ SessionOwnerLease→ RemoteRunQueue (顺序)→ Heartbeat 续租→ 回写节点 output/error
对象职责失败语义
ExternalAgentSession持有远程连接与生命周期断连 → 进入 stale，等待 recovery。
SessionOwnerLease单写所有权 + TTL租约过期 → 释放并可被接管。
RemoteRunQueue + requestId顺序执行 + 幂等重复投递被 requestId 吸收。
RemoteRunHeartbeat存活探测心跳丢失 → 取消传播 + 重试或失败上报。

10. 自学习闭环

让系统从「记住事实」进化到「沉淀方法」，但保留人类在关键回路（候选必经审核）。

Task Outcome→ Reflection→ SkillCandidate→ SkillReview（人工）→ PublishedSkill
分层记忆：session context（短期）/ persistent facts（事实）/ procedural skills（过程）/ reflection candidate（反思）。 Dreaming 与 Skill 候选共用「Review-First」不变量——只产候选，不静默改库。

技能渐进式加载

先注入 SkillIndexEntry（短索引），模型决策命中后再加载完整 SKILL.md，降低 prompt 体积与跨域误触发。

11. 运行时与可靠性（Day-2）

与领域建模分离的独立专章，避免运维细节污染领域层，但交叉引用。

可观测性
  • 领域事件天然双写 trace/timeline。
  • 统一 correlationId 链：taskId / runId / heartbeatRunId 串成一条 trace。
  • 「3am 只看日志能否定位」测试。
失败与降级
  • 每个跨上下文调用定义超时矩阵。
  • orphan artifact / 半状态 → 后台对账。
  • 外部 Agent：lease 过期 / 心跳丢失 / stale recovery 状态机。
自动化护栏
  • Heartbeat Pulse 确定性归约，不调 LLM、不建任务。
  • Dispatch 升级为任务需速率限制 + 熔断。
  • 失控保护：单位时间任务/外发上限。
恢复
  • 事件可重放 → Session/Task 状态可重建。
  • 快照 + 事件混合恢复，降低重放成本。

12. §11「行业最佳实践」是否合理 / 能否作为新方案

结论：方向合理，可作为新设计的骨架，但不可原样作为方案。 需要三处修正：①外部对标降权为参考；②补三大缺失契约（已在 §6/§7/§11）；③新对象按三档分级 + 迁移代价。
§11 主张评估本方案处理
稳定内核 / 用户态分层合理采纳为主轴，补 Event Store + Capability（§3）。
MCP=工具 ACL / ACP=Agent ACL 分层合理升为设计原则 P5，文档与命名禁止互相替代。
外部 Agent 持久会话合理采纳，落为显式状态机（§9）。
Hook/Guardrail 升级为领域对象合理但需克制采纳目标，但按「两个消费者」门槛引入，先用字段+事件。
Steward Agent 绑定上下文合理采纳为原则 P3（默认零权限 + 按边界授予）。
引用 Hermes / OpenClaw / acpx / Qualixar OS 作为「业界事实」可验证性不足全部降权为「灵感参考」，加限定语，不作权威依据。
§11.4 一次性提议 ~15 个新对象过度建模风险三档分级，多数标 演进/暂缓。

13. 演进路线（现状 → 目标 → 代价）

优先级变更现状 → 目标代价 / 风险
现在 P0MCP/ACP 边界澄清命名混用 → 文档与代码命名固定职责纯文档/命名，低成本高收益。
现在 P0一致性与事件契约固化隐式 → §6/§7 显式契约 + 对账需补对账任务与 schema 版本，中等。
现在 P0策略对象测试矩阵隐式于 service → 规则级单测PermissionRule / HeartbeatPulse / ChannelSpecialization，中等。
演进 P1外部 Agent 持久会话HTTP 式调用 → Session+Lease+Queue+Heartbeat真实运行时工程，需状态机，较高。
演进 P1Agent 上下文绑定 + 技能渐进加载全量工具/技能 → 按边界授予 + 索引加载需路由与 ranking，中等。
演进 P1自学习闭环Dreaming 候选 → Reflection→Candidate→Review→Published需审核 UI 与技能库治理，中等。
暂缓 P2内核范式：Event Store 为源 + Capability 令牌Task 投影/询问式 → 事件溯源/能力安全较大范式迁移，待两个真实驱动力再启动。
采纳建议：先做三项 P0（皆为契约/文档/测试，低风险高价值），让设计「可被增量采纳」；P1 按需推进；P2 作为标注清晰的目标态，避免一次性大重构。
