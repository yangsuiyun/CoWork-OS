2. 设计基准（承接目标方案）

本文继承目标方案的 7 条原则，并把它们翻译为可验证的技术约束。

原则技术约束（可验证）
P1 内核稳定 / 能力易变内核包不得 import 用户态包；用依赖检查（如 dependency-cruiser / eslint-boundaries）在 CI 强制。
P2 执行即事实，事件即真相所有状态变更必须先写 event_log；读模型只能由投影器写入，禁止直接 UPDATE 业务状态。
P3 默认零权限，按边界授予Agent 启动时 capability set 为空；工具执行必须出示有效 capability token。
P4 单向依赖（DAG）模块依赖图无环；跨上下文只经 event bus 或 repository 接口。
P5 边界即防腐层MCP / ACP 各有独立 adapter 包，核心域不出现外部协议类型。
P6 失败显式每类失败有 typed error + 补偿/对账任务；无静默 catch-return-null。
P7 人类在关键回路现实副作用 / 记忆固化 / 技能发布须经 ApprovalRequest 或 Review 状态机。

3.2 L2 容器视图（服务端 + 瘦客户端）

Web / 移动 / CLI 客户端

API Gateway / BFF
TLS 终结、鉴权(JWT/session)、租户隔离、限流；HTTP(命令/查询) + WebSocket(事件推送)。

Capability Services
编排、技能、网关、记忆；可独立伸缩/崩溃重启的用户态服务。
Connector / Agent Runner
MCP / ACP / channel adapter、computer-use 执行器，沙箱隔离的服务端子进程。
数据层
PostgreSQL（event_log + 读模型）+ pgvector（向量）+ 对象存储（工件文件）。
瘦客户端原则：客户端只做「渲染 + 发命令 + 订阅事件」，所有领域逻辑、状态、安全都在服务端——这是「多端可直接使用」且行为一致的前提，新增端只需实现同一套 API/WS 协议。

4. 进程与部署拓扑（B/S）

服务端集中承载内核与能力层；浏览器 / 移动 / CLI 多端瘦客户端经 HTTPS + WebSocket 接入。进程隔离用于故障隔离 + 权限隔离；无状态服务可水平伸缩，事件存储为单一事实源。

部署单元形态职责崩溃影响 / 伸缩
客户端Web SPA / 移动(PWA 或套壳) / CLI渲染投影、发命令、订阅事件仅本端；重连后从读模型恢复
API Gateway / BFF无状态服务TLS、鉴权、租户隔离、限流、WS 网关无状态 → 多副本 + 负载均衡
Kernel Service有状态(写)服务事实源、命令处理、事件存储单写者；崩溃不丢数据可重放；按租户/工作区分片扩展
Capability Services无状态 / 可重启编排 / 技能 / 网关 / 记忆独立伸缩；崩溃不波及内核，断点续跑
Connector / Agent Runner沙箱子进程 / 容器MCP / ACP / channel、computer-use熔断重连；租约失效释放
Local Agent Runner（可选）装在用户设备上的轻量代理代表用户操作其本地电脑 / 浏览器 / 文件断线则本地能力降级（见 §20）
关键性质（不变）：内核仍是事件溯源，任何服务崩溃都不产生半状态——未提交命令视为未发生，已提交事件可重放。B/S 下额外获得「服务端重启不影响多端会话连续性」。
本地能力的核心取舍：「操作用户本机」类能力（computer-use、本地文件、本地 shell）在纯服务端无法直达用户设备。两条路：①Local Agent Runner（装在用户电脑上，与服务端建反向安全通道）；②服务端云沙箱（语义从"操作我的电脑"变为"操作云端工作空间"）。详见 §20 功能影响分析。

4.1 部署形态

self-host 单机:  docker compose（gateway + kernel + workers + postgres + minio）
团队 / 云:       k8s（gateway 多副本 · kernel 按租户分片 · postgres 主从 · 对象存储）
桌面单机(后续):  同一服务端内嵌打包（Tauri/Electron 壳本地起 server）→ 复用 B/S 代码

5. 技术栈选型（B/S，含备选与理由）

关注点推荐理由备选
前端（多端同构）React + Vite + TypeScript，响应式一套代码自适应手机/电脑；与后端共享类型。Vue 3 / SvelteKit
移动端PWA 优先（可安装/推送），需上架时 Capacitor 套壳复用 Web 代码最快覆盖多端；原生能力按需补。React Native（投入更大）
响应式策略断点自适应 + 移动端精简工作台手机与电脑信息密度差异大，需分别裁剪（见 §20）。独立移动 App
服务端Node.js + TypeScript + Fastify前后端同语言/同类型；适配 LLM/MCP；高吞吐。NestJS（更重）/ Go（性能）
实时推送WebSocket（事件流）+ 断线重连 + 游标续传多端实时一致；移动后台/弱网需重连补偿。SSE（单向回退）
鉴权OIDC/OAuth2 + 短期 JWT + 刷新令牌多端统一身份；支持第三方登录。session cookie（同源 Web）

工件存储对象存储（S3 兼容 / MinIO 自托管）大文件不进库；多端经签名 URL 访问。服务端文件系统（单机）
本地能力Local Agent Runner（gRPC/WS 反向通道）服务端模型下仍能操作用户本机（见 §20）。云沙箱（无需装客户端）
外部协议MCP（工具/数据）、ACP/A2A（Agent）各自防腐层（P5）。—

说明：选型从「B/S 多端最优」反推。领域 / 事件 / 一致性设计与传输无关——后续做桌面单机版时，把同一服务端内嵌打包、前端复用即可，不动核心。

6.1 内核对外的唯一接口（窄契约）

interface Kernel {
  // 写：所有变更经命令，返回已提交事件（或 typed error）
  dispatch(cmd: Command): Promise<Result<CommittedEvent[], DomainError>>;
  // 读：只读投影，永不返回可变领域对象
  query<T>(q: Query<T>): Promise<T>;
  // 订阅：用户态通过事件流协作（at-least-once + 幂等消费）
  subscribe(filter: EventFilter, from: Cursor): AsyncIterable<CommittedEvent>;
}
边界纪律：用户态只能通过 dispatch/query/subscribe 触达内核。任何「直接读写内核数据库表」都是架构违例，CI 依赖检查拦截。

7. 领域模型详化

把目标方案的聚合落为带完整属性、不变量与命令/事件的规格。仅示核心域，支撑域同构。

7.1 Task 聚合

type Task = {
  id: TaskId; workspaceId: WorkspaceId;
  origin: 'manual'|'cron'|'hook'|'api'|'channel';
  canonicalPrompt: string;        // rawPrompt→userPrompt→prompt 解析后的不可变定义
  status: TaskStatus;             // 由事件投影，非直接写
  parentTaskId?: TaskId; sessionId?: SessionId;
  agentConfig: AgentConfigRef; budget?: Budget; risk?: RiskTier;
  version: number;               // 乐观并发

// 命令 → 不变量校验 → 事件
Commands: CreateTask | PlanTask | StartTurn | RequestApproval | ResolveApproval
        | AppendArtifact | CompleteTask | FailTask | CancelTask | DeriveChildTask
Invariants:

  - INV-3 子任务必须携带 parentTaskId 且共享 workspace 治理

8. 数据与事件存储（Schema）

8.1 事件存储（单写者：内核）

-- append-only，永不 UPDATE/DELETE
CREATE TABLE event_log (
  global_seq   INTEGER PRIMARY KEY AUTOINCREMENT,  -- 全局顺序
  stream_id    TEXT NOT NULL,      -- 聚合实例，如 task:<id>
  stream_seq   INTEGER NOT NULL,   -- 聚合内顺序（乐观并发）
  type         TEXT NOT NULL,      -- TaskCreated 等
  schema_ver   INTEGER NOT NULL,   -- payload 版本
  payload      TEXT NOT NULL,      -- JSON
  actor        TEXT NOT NULL,
  correlation_id TEXT, causation_id TEXT,
  occurred_at  TEXT NOT NULL,
  UNIQUE(stream_id, stream_seq)        -- 并发写冲突 → 重试命令
);
CREATE TABLE outbox (                  -- 事务性外发，保证至少一次投递
  global_seq INTEGER PRIMARY KEY, dispatched INTEGER DEFAULT 0
);

8.2 读模型（投影，可重建）

-- 由 Projector 消费 event_log 写入；可 DROP 后从事件重放重建
CREATE TABLE rm_tasks (id, workspace_id, status, title, updated_seq, ...);
CREATE TABLE rm_timeline (task_id, seq, kind, summary, occurred_at);
CREATE TABLE rm_artifacts (id, task_id, path, sha256, mime, size, created_at);
CREATE TABLE rm_memory_vec (id, scope, kind, embedding, text, source_event);
关键模式：同库事务内「append event_log + 写 outbox」原子提交（Transactional Outbox），投影与外发都从 outbox/event_log 异步驱动；这把「Artifact 落盘成功但事件写失败」这类半状态从根上消除。

8.3 迁移与版本

  • 读模型迁移：直接 DROP + replay，无需手写数据迁移（事件是真相）。
  • 事件 schema 演进：只增不改；老事件由 upcaster（schema_ver → 最新）在读取时升级。

9. 事件总线与契约

9.1 投递语义

  • at-least-once：消费者必须幂等（按 global_seq 去重 / 按业务键幂等）。
  • 有序：单 stream 内严格有序；跨 stream 用 global_seq 全序。
  • 游标：每个消费者持久化自己的 cursor，崩溃后从断点续读。

9.2 事件目录（节选，含 payload 契约）

TaskCreated v1 { taskId, workspaceId, origin, canonicalPrompt, correlationId }
ApprovalRequested v1 { taskId, kind: 'tool'|'data_export'|'shell'|..., risk, context }
ArtifactCreated v1 { artifactId, taskId, path, sha256, mime, size }
TaskCompleted v1 { taskId, result, metrics }
RemoteAgentDelegated v1 { sessionId, agentId, requestId, scope }  演进
契约纪律：事件名 + payload schema 进版本库并做契约测试；新增字段必须可选；删除字段须先 deprecate 一个版本。消费者按 schema_ver 经 upcaster 读取。

10. 一致性、事务与恢复

场景机制实现要点
聚合内强一致单库事务命令处理在一个 Postgres 事务内校验不变量 + append 事件。
跨聚合最终一致Transactional Outbox + 投影事件提交后异步驱动投影与外发，消费幂等。
并发写冲突乐观并发UNIQUE(stream_id, stream_seq) 冲突 → 重新加载 + 重试命令。
外部副作用（发送/部署）幂等键 + saga 补偿requestId 幂等；失败走补偿事件而非回滚已发生的现实动作。
崩溃恢复snapshot + replay定期对热聚合快照；恢复 = 加载快照 + 重放其后事件。
对账（reconciliation）后台扫描orphan artifact / 未投递 outbox / stale session 周期对账修复。
 
// 命令处理统一骨架（伪代码）
async function handle(cmd) {
  const agg = await load(cmd.streamId);          // snapshot + replay
  const events = agg.decide(cmd);                 // 纯函数：校验不变量→产出事件
  await tx(() => { append(events); enqueueOutbox(events); }); // 原子
  return ok(events);                             // 投影/外发异步发生
}

11. 鉴权、多租户与安全内核

B/S 新增攻击面：系统从「单机本地」变为「网络暴露的服务端」，必须新增传输安全、身份认证、授权与租户隔离——这是 B/S 相对 local-first 最大的安全增量。

11.0 鉴权与租户隔离（B/S 新增）

层机制
传输全程 TLS；WebSocket 同源 + token 鉴权。
认证 AuthNOIDC/OAuth2 登录 → 短期 JWT + 刷新令牌；多端共享同一身份。
授权 AuthZ每个命令/查询校验 user→workspace 角色与能力；事件订阅按 workspace/tenant 过滤，禁止越权读他人事件流。
租户隔离数据带 tenant_id/owner_id；行级隔离（Postgres RLS 或强制查询条件）；工件签名 URL 限范围。
审计登录、授权变更、data_export、审批决策均落事件，可追溯。
与能力模型的关系：身份(AuthN) 解决「你是谁」，能力令牌(§11.1) 解决「这次执行能做什么」。命令先过 AuthN/AuthZ，再由 CapabilityIssuer 在身份/工作区范围内授予最小能力。

11.1 能力令牌（Capability Token）

type Capability = {
  subject: TaskId|AgentId; resource: 'fs.write'|'net'|'shell'|'data_export'|'tool:<name>';
  scope: { paths?: Glob[]; domains?: string[]; workspaceId: WorkspaceId };
  constraints?: { maxCalls?: number; expiresAt?: string };
  revocable: boolean;            // hard-deny 对应「不可铸造」的能力
};
  • 签发：Task/Agent 启动时由 CapabilityIssuer 依据 workspace 能力 + bounded context 授予最小集合。
  • 校验：工具执行前必须出示令牌；无令牌或越界 → deny。
  • 吊销：审批拒绝 / 风险升级 → 立即吊销，正在进行的调用被中断。

11.2 确定性 Hook 管线

PreToolUse Hook→ Capability 校验→ PermissionRule 匹配→ allow / deny / ask→ 执行→ PostToolUse Hook 
decision = PreHooks.run(ctx)               // 确定性、可审计，可直接 deny/transform
        ?? Capability.check(ctx)
        ?? PermissionRules.match(ctx)       // allow|deny|ask
if (decision === 'ask') emit ApprovalRequested(...)  // 人类在回路 (P7)
不可覆盖集：危险命令、hard guardrail、task restriction、显式 deny、data_export 不被 mode/fallback 覆盖；这些规则有规则级单测矩阵。

12. 编排与外部 Agent 实现

12.1 编排图引擎

OrchestrationGraph 是 DAG：节点 = 子任务或委派目标；引擎按依赖就绪调度，节点状态由事件回写。本地子代理与远程 Agent 用同一节点抽象，区别仅在 dispatchTarget。

12.2 外部 Agent 持久会话（actor + 租约） 演进

ExternalAgentSession (actor)
  ├─ SessionOwnerLease { ttl, holder }     // 单写所有权，过期可接管
  ├─ RemoteRunQueue [ { requestId, ... } ] // 顺序执行 + 幂等
  ├─ Heartbeat (interval)                  // 丢失→ SessionStale 事件
  └─ StreamEvents → 规范化 → NodeUpdated   // 回写编排图

状态机: Discovered → Authorized → Open → Running ⇄ Idle → (Stale → Recover) → Closed
防腐层：adapter-acp 把外部协议消息翻译为内核事件；核心域不出现 ACP 协议类型。MCP 同理走 adapter-mcp，仅暴露「工具调用 + 数据访问」语义。

14. 通道网关

  • 统一入口：每个通道一个 adapter-channel-* 子进程，归一化为 InboundMessage → 命令。
  • specialization 路由：新任务按 specialization 选 workspace/role；已有任务 follow-up 粘住原语境（不中途换身份）。
  • 共享记忆 opt-in：群组/公共通道默认不注入共享记忆，除非 specialization 显式允许。
  • 会话绑定：ChannelSession 以 taskId 绑定当前任务，回流为 TaskEvent。

15. API 契约（多端）

15.1 多端客户端 ↔ 服务端（HTTP + WebSocket）

# 命令与查询：HTTP（带 Bearer JWT）
POST /v1/commands            { type, payload }   → Result<CommittedEvent[]> | DomainError
GET  /v1/query/:name?...                          → ReadModel（按身份/租户过滤）

# 实时事件：WebSocket（鉴权 + 按 workspace 过滤 + 游标续传）
WS   /v1/stream?cursor=...   ⇄ { event }          // 断线重连从 cursor 续读，幂等
多端一致：Web / 移动 / CLI 共用同一套命令/查询/事件协议，行为天然一致；新增端只需实现协议。移动端弱网/后台需处理 WS 重连 + cursor 续传（见 §20）。

15.2 外部 API（Managed Sessions，给程序化调用）

POST /v1/sessions            → 创建 durable run（底层映射到 Task / AgentTeamRun）
GET  /v1/sessions/:id        → 状态（读模型投影）
GET  /v1/sessions/:id/events → SSE 事件流
POST /v1/sessions/:id/cancel → 取消（命令）
不变量：ManagedSession 不直接执行，必映射到 Task/AgentTeamRun；对外只暴露命令 + 事件投影，与内核保持同一契约风格。

20. B/S 模式功能影响分析（哪些功能会有问题）

从 local-first 桌面转为 B/S 服务端后，领域逻辑全部可平移，但「依赖用户本机环境」与「桌面原生/移动端」的能力会受影响。下表按严重度列出受影响功能、问题本质与缓解方案。

头号问题（必须先决策）：CoWork OS 的「computer use / 浏览器自动化 / 本地文件 / 本地 shell」本质是操作用户自己的电脑。纯服务端摸不到用户设备，这是 B/S 化最大的功能断点。推荐方案：把 Local Agent Runner 作为一等组件（用户设备上装轻量代理，与服务端建反向安全通道），并在产品上明确区分两种工作空间语义——「☁️ 云端工作空间」（服务端沙箱，开箱即用）与「💻 我的电脑」（需 Local Runner，操作本机）。

20.1 受影响功能总表

功能B/S 下的问题严重度缓解方案
Computer Use / 浏览器自动化服务端无法控制用户本机的浏览器/桌面高Local Agent Runner 在本机执行；或云端无头浏览器（语义=云端而非本机）
本地文件系统 / Workspace 本地路径workspace.path 指向用户磁盘，服务端不可达高云端工作空间(对象存储)；或 Local Runner 映射本地目录；workspace 语义二分
本地 Shell / 终端服务端 shell ≠ 用户机；且共享服务端 shell 有越权风险高每租户隔离的云沙箱；本机 shell 经 Local Runner + 能力令牌治理
本地 MCP 工具 / 本地连接器装在用户机上的工具服务端调不到中Local Runner 转发；或迁为服务端托管连接器
本地优先 / 隐私数据上云，敏感内容离开设备（原卖点）中自托管部署 + 区域化存储 + 端到端加密(后续) + data_export 审批
离线使用B/S 强依赖网络中客户端只读缓存 + 离线排队命令，联网后重放
桌面原生集成（托盘/全局热键/本地通知/剪贴板/拖拽）浏览器沙箱受限中PWA 通知 + 文件 API；强需求项交由 Local Runner / 桌面壳(M5)
长任务结果送达WS 断开（移动后台/锁屏）会漏掉完成事件中cursor 续传补拉 + Web Push/系统推送通知
大工件预览 / 流式输出大文件与 token 流经网络，延迟/带宽低签名 URL 直传对象存储 + 分块/缩略 + 流式增量
多租户共享服务端一个租户的工具/shell 不能触达另一个中每租户沙箱隔离 + RLS + 能力令牌限定 workspace

20.2 移动端 + 电脑端「同时支持」的专项影响

前端要一套代码同时服务手机与电脑，差异不只是屏幕宽度，还包括能力与交互：

布局与信息密度
编排图 / 时间线 / 多面板工作台在手机上放不下 → 移动端改为单列 + 抽屉 + 摘要优先，复杂可视化降级为只读概览或延后到平板/桌面。
实时连接
移动端后台会杀 WS/限网 → 必须 cursor 续传 + 推送通知；桌面端长连接稳定，可全量实时。
输入与文件
手机无终端、文件系统受限、以触控为主 → 终端/批量文件操作在移动端隐藏或转交云沙箱；桌面端保留完整能力。
本地能力
手机几乎无法装 Local Runner 操作"本机" → 移动端默认只用云端工作空间；"操作我的电脑"主要在桌面端可用。
通知与后台
移动用 Web Push/系统推送做异步送达；桌面可用 PWA 通知或桌面壳通知。
性能
移动端算力/内存有限 → 大列表虚拟化、按需加载、减少前端计算（领域逻辑本就在服务端，利好移动端）。
结论：采用「响应式同构 + 能力分级」策略——同一套 React 代码，按视口 + 设备能力裁剪：移动端聚焦「监控/审批/对话/简报」，桌面端解锁「完整工作台/编排/终端/本机操作」。这与 B/S「领域逻辑在服务端」天然契合，移动端因此可以很薄。

20.3 移动端 vs 桌面端功能矩阵

图例：✓ 可用 · ◐ 降级（精简/只读/转云沙箱） · ✕ 隐藏（该端不提供）。一套响应式代码按视口 + 设备能力裁剪。

功能移动端桌面端说明
对话 / 提任务✓✓核心入口，两端一致。
任务监控 / 时间线◐✓移动端摘要优先、单列；桌面端完整事件流。
审批决策✓✓高频移动场景（随时批/拒），重点优化。
简报 / 通知✓✓移动端走 Web Push / 系统推送。
Everything Workbench（工件）◐✓移动端只读预览/缩略；桌面端完整编辑。
编排图可视化◐✓移动端只读概览；桌面端交互式 DAG。
终端 / Shell✕✓移动端隐藏；桌面端云沙箱或本机(Runner)。

实时流式输出◐✓移动端弱网/后台降级为补拉；桌面端全量实时。
读法：移动端聚焦「监控 / 审批 / 对话 / 简报」这条高频轻量主线；桌面端解锁「完整工作台 / 编排 / 终端 / 操作本机」。因领域逻辑都在服务端，移动端可以做得很薄。

20.4 Local Agent Runner 反向安全通道（时序）

解决头号难点「服务端如何操作用户本机」：Runner 装在用户电脑上，仅出站连服务端（无需开放入站端口/防火墙），建立持久反向隧道，受能力令牌 + 本地策略双重治理。

Client(浏览器/桌面)     Server(Kernel + Permission)        Local Agent Runner(用户电脑)
     │                        │                                   │
     │                        │   ① 启动：出站连接 + 鉴权(mTLS/OIDC)      │
     │                        │ ◀──────────────────────────────────│  反向隧道(仅出站)
     │                        │   ② 注册本机能力(fs / shell / browser)  │
     │                        │ ◀──────────────────────────────────│
     │ ③ 发命令“操作我的电脑”      │                                   │
     │ ──────────────────────▶│                                   │
     │                        │ ④ AuthZ + 签发能力令牌(限本设备/限路径/TTL) │
     │                        │   ⑤ LocalActionRequested(token) 经隧道下发 │
     │                        │ ──────────────────────────────────▶│
     │                        │                                   │ ⑥ 校验 token + 本地策略
     │                        │                                   │   (高风险动作可弹本机确认)
     │                        │                                   │ ⑦ 本地执行(读写文件/shell/浏览器)
     │                        │   ⑧ 流式结果 / 事件 ◀────────────────│
     │                        │ ⑨ 规范化为 TaskEvent + 投影 + 审计       │
     │ ⑩ WS 推送时间线更新       │                                   │
     │ ◀──────────────────────│                                   │
     │                        │   ◇ 心跳续租；断线→能力降级 + SessionStale  │
为何反向出站
用户机多在 NAT/防火墙后；出站隧道免开端口，部署摩擦最低，攻击面更小。
安全治理
能力令牌限定「本设备 + 路径/域名范围 + TTL + 可吊销」；本地策略可对危险动作二次确认；全程落审计事件。
可靠性
租约 + 心跳；断线进入 stale，本地能力优雅降级，恢复后续跑（复用 §12 外部会话机制）。
语义边界
有 Runner = 可选「💻 我的电脑」；无 Runner（如手机）= 仅「☁️ 云端工作空间」，行为清晰不混淆。

20.5 不受影响（平移即可）

Task 执行与编排、事件溯源时间线、记忆与自学习、审批与权限治理、外部 Agent 协作、通道网关、Managed Sessions、可观测性——这些是传输无关的领域能力，B/S 下逻辑不变，反而因「服务端常驻」更适合长任务与多端协作。
