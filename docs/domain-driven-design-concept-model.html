1. 执行摘要

CoWork OS 的核心业务不是“聊天机器人”，而是一个本地优先、GUI-first 的个人/团队 AI 工作操作系统。 它把用户请求建模为可执行、可观察、可恢复、可治理的 Task，再通过 Agent Runtime、工具权限、记忆学习、 工作台工件、通道网关和自动化循环，把工作从“对话”推进到“真实产出”。

核心域判断：项目最有业务差异化的核心域是“可治理的代理任务执行与多代理协作”。围绕它形成三个强支撑域： Workflow Intelligence（记忆、Heartbeat、Dreaming、Suggestions）、Everything Workbench（工件工作台）和 Integration/Gateway（通道与连接器）。

领域分层

DDD 分类 领域 业务含义

Supporting Domain Memory & Workflow Intelligence、Everything Workbench、Channel Gateway、Skills Runtime、MCP Connectors 让执行过程具备长期记忆、可复用技能、远程消息入口、生成工件和外部系统连接能力。
Generic Domain Profiles、Settings、Update、Telemetry、Reports、Native Helpers、Security Storage 桌面应用运行所需的配置、持久化、系统能力、打包和通用工程支撑。

3. 统一语言

以下术语建议作为项目内产品、设计、后端、前端、测试共同使用的统一语言。

概念 定义 常见字段/状态 典型行为
Workspace 用户工作的本地边界，绑定文件路径、权限、记忆、策略和任务。 id, name, path, permissions, lastUsedAt 创建、切换、更新权限、同步 `.cowork/` 工作区上下文。
Task CoWork OS 最核心的执行单元，代表一次用户目标、自动化目标或代理派生目标。 status, workspaceId, agentConfig, parentTaskId, sessionId, terminalStatus 创建、计划、执行、暂停、继续、取消、完成、重试、派生子任务、记录结果。
SessionRuntime 任务会话运行时状态拥有者，管理权限状态、检查清单、快照、恢复和任务投影。 checklist, permissionState, snapshot, verificationState 准备 turn、协调工具调用、持久化快照、处理 follow-up、恢复执行。
Turn 一次模型-工具-结果循环中的执行步，由 TurnKernel 与 ToolScheduler 处理。 mode, stepId, tool calls, result envelope 构造 prompt、选择工具、批调度、产出事件、更新运行态。
Agent 具备角色、模型、工具限制、记忆策略和执行能力的工作执行者。 AgentConfig, AgentRole, WorkerRole, capability 执行任务、协作、验证、研究、综合、通过角色策略受限。
ManagedAgent 可复用、可版本化的代理定义，面向 Agents Hub 和控制平面。 ManagedAgentVersion, runtimeDefaults, approvalPolicy, memoryConfig 创建、更新、归档、发布、测试、预览、生成 ManagedSession。
ManagedSession 托管代理的一次持久运行资源，API 面向稳定 run identity，底层映射到 Task。 agentId, environmentId, status, backingTaskId, events 创建、取消、恢复、发送事件、镜像 TaskEvent。
Artifact 任务创建或更新的文件产物，例如 DOCX、XLSX、PPTX、HTML、PDF、代码或数据文件。 taskId, path, mimeType, sha256, size 登记、预览、打开、全屏工作、跟随 follow-up 刷新。
Approval / Permission 工具执行前的治理机制，决定 allow、deny 或 ask，并可持久化规则。 PermissionRule, PermissionMode, ApprovalRequest, securityContext 评估工具请求、生成审批、保存规则、阻断危险操作、处理 data export。
Memory 长期上下文和工作经验的持久来源，包括偏好、约束、观察、工作流模式和关系记忆。 workspaceId, type, content, embedding, privacyState 捕获、压缩、检索、注入 prompt、归档、策展、主题包化。
Heartbeat 后台自动化的信号就绪层，用 Pulse/Dispatch 决定何时提示、何时升级为任务。 HeartbeatSignal, PulseResult, Dispatch, RunRecord 合并信号、延迟压缩、触发建议、创建任务、触发 Dreaming。
Dreaming 后台记忆健康检查与候选提案机制，提出可审核的记忆维护变更。 DreamingRun, DreamingCandidate, evidenceRefs, status 收集证据、生成候选、审核、应用到 Curated Memory。
Channel 远程消息入口，例如 WhatsApp、Telegram、Slack、Discord、Email 等。 type, securityConfig, status, ChannelSession, ChannelMessage 配对、鉴权、会话管理、消息入站、命令路由、进度回传。
Skill 可附加到任务上的执行指导包，提供上下文和运行时指令，但不替换原始任务。 SkillApplication, trigger, contextDirectives, parameters 路由、应用、参数收集、限制工具、指定产物目录。
MCP Connector 外部系统连接器和工具服务，扩展 CoWork 对 SaaS/企业系统的操作能力。 server, tools, oauth, status, allowlist 连接、鉴权、暴露工具、隧道转发、权限过滤、错误通知。

本节将对象按 DDD 类型归类。由于项目规模较大，“所有领域对象”按业务概念族合并呈现，而不是列出每个 DTO 或 UI-only 类型。

6. 上下文详细建模

6.1 任务执行上下文

职责：维护从用户目标到执行结果的完整生命周期。它是全系统的核心上下文。

对象 属性 行为 不变量/规则
Task title、canonical prompt、status、workspaceId、agentConfig、budget、lineage、risk、outputs create、update status、spawn child、continue、branch、complete、record outcome 任务必须属于 Workspace；新任务需保留 canonical prompt；终态任务不能被 late resume 重新打开。
TaskStrategySnapshot intent、conversationMode、executionMode、taskDomain、workflowMode、confidence 描述路由策略，指导执行模式、预检和模型选择 策略可以指导执行，但不能覆盖用户原始意图。
SessionRuntime taskProjection、checklist、permissionState、recoveryState、outputState prepare turn、apply follow-up、persist snapshot、recover、project UI state SessionRuntime 是任务会话状态单一拥有者；权限状态必须快照持久化。
TaskEvent taskId、seq、actor、type、payload、status、timestamp append、page、detail、semantic projection 事件按 taskId 和序号形成追加式时间线。

6.2 代理编排上下文

职责：把复杂任务分配给多个角色、模型或远程代理，并收敛为可理解结果。

对象 属性 行为 规则
AgentRole displayName、description、capabilities、tool policy、heartbeatPolicy 作为任务角色、被 mention、参与团队、触发后台自动化 角色工具范围应小于或等于 Workspace/Task 权限。
WorkerRole researcher、implementer、verifier、synthesizer 限制子代理职责和工具范围 验证者应使用独立判定合同输出 verdict/report。
AgentTeamRun team、members、items、thoughts、childTasks、status 创建团队运行、分配工作、聚合输出 团队运行仍以 Task/TaskEvent 为执行事实源。
DelegationGraph run、node、event、parent-child relation 统一 spawned agents、multitask lanes、workflow phases、ACP tasks 所有委派工作应进入同一图模型，方便追踪和恢复。

6.3 托管代理上下文

职责：把底层执行能力产品化为可配置、可版本、可审计的 Agents Hub 资源。

聚合/对象 属性 行为 规则
ManagedAgent name、status、currentVersion、template、audit create、update、archive、convert from role/profile 自身不直接执行，必须通过 ManagedSession 映射到 Task。
ManagedAgentVersion systemPrompt、model、executionMode、runtimeDefaults、studio metadata 定义一次运行使用的代理行为和默认预算 runtimeDefaults.maxTurns 是显式上限；未设置则继承主任务默认行为。
ManagedEnvironment workspaceId、tool policy、MCP allowlist、file paths、credential refs 配置本地执行环境、工具范围和文件上下文 renderer-facing 读取必须隐藏 credentialRefs 和 managedAccountRefs。
ManagedSession agentId、environmentId、surface、status、backingTaskId、event history create、cancel、resume、sendEvent、mirror task events API-facing durable run 是 ManagedSession，执行 worker 是 Task。

6.4 权限与安全上下文

职责：在高能力代理和本地/外部资源之间提供清晰的审批、阻断和审计边界。

对象 属性 行为 规则
PermissionRule source、effect、scope、metadata match tool/domain/path/command/mcp_server 更具体的规则优先；同等时 session > workspace > manifest > profile > legacy > mode。
PermissionEvaluationResult decision、reason、matchedRule、suggestions、scopePreview 给工具执行返回 allow/deny/ask 硬阻断不能被后续规则覆盖。
ApprovalRequest type、description、details、status、requestedAt request、approve、deny、resolve data_export、location_access 等高风险类型保持独立审批语义。
FileProvenanceRecord path、sourceKind、trustLevel、sourceLabel、recordedAt 标记导入文件来源，驱动不可信内容与导出审批上下文 从 channel/clipboard/import 进入的文件默认需要 provenance-aware 处理。

6.5 记忆与 Workflow Intelligence 上下文

职责：让系统在长期使用中保留偏好、事实、约束、关系、承诺和可复用工作流经验。

对象 属性 行为 规则
Memory workspaceId、type、content、summary、embedding、privacy capture、compress、search、inject、archive 敏感信息模式应过滤；压缩/检索应遵循预算和隐私状态。
MemoryObservation facts、concepts、filesRead、tools、sourceEventIds 从任务证据生成结构化观察 观察应保留来源，便于后续 recall 和 Dreaming 审核。
RelationshipMemory identity、preferences、context、history、commitments 存储个人连续性、承诺生命周期、偏好反馈 共享频道记忆隔离，除非明确可信/开启共享记忆。
KnowledgeGraph nodes、edges、temporal validity、source evidence 链接人、公司、项目、文件、事件和事实 时间感知边需要保留有效期，防止历史事实覆盖当前状态。
DreamingRun/Candidate trigger、scope、evidence、candidate action、target、status 收集证据、生成记忆维护候选、审核应用 Dreaming 只提案，不静默改写记忆。

6.6 Heartbeat 与自动化上下文

职责：管理后台“何时应该思考/提醒/执行”的节奏，避免把所有环境变化都变成任务。

对象 属性 行为 规则

7. 跨上下文关系

Task 是执行事实源

ManagedSession、ChannelSession、HeartbeatRun、AgentTeamRun 最终都会关联到 Task 或 TaskEvent。Task 是所有真实执行的汇聚点。

  • ChannelSession 通过 taskId 绑定远程对话的当前任务。
  • ManagedSession 创建 exactly one backing Task。
  • Heartbeat Dispatch 创建的任务携带 heartbeatRunId。
  • Artifact、Approval、TaskEvent 都以 taskId 作为归属。

Workspace 是治理边界

Workspace 将本地路径、权限、记忆、工作区 kit、通道 specialization 和托管环境连接起来。

  • WorkspacePermissions 决定读写、删除、网络、Shell 能力。
  • Workspace-local permission rules 可落库并镜像到 `.cowork/policy`。
  • Memory、Observation、Heartbeat、ManagedEnvironment 均按 workspace 归属。

Skill 是附加上下文

SkillApplication 影响执行上下文和工具暴露，但不修改 canonical task text。

  • Skill 可由 slash、planner、model 或 explicit hint 触发。
  • contextDirectives 可提供 allowedTools、toolRestrictions、modelHint、artifactDirectories。
  • 任务定义始终由 rawPrompt/userPrompt/prompt 的 canonical 顺序决定。

Memory 是学习事实源

任务完成、Chronicle、Heartbeat、Relationship Agent、Inbox 等都可能产生记忆证据；Dreaming 只产出候选。

  • MemoryService 捕获、压缩、检索和注入。
  • MemoryObservation 将任务证据结构化。
  • DreamingCandidate 需要被审核后才会应用。

10. 战略设计视角

核心域

可治理代理执行平台 是项目竞争力核心：Task Runtime、Agent Orchestration、Permission Engine、Managed Agents 共同形成 “从自然语言目标到可审计执行产物”的业务闭环。这里应保持高内聚，避免被 UI 细节、连接器细节或具体 provider 细节污染。

支撑域

Memory/Heartbeat/Dreaming、Everything Workbench、Gateway/Connectors、Skills 是支撑域。它们增强核心域的长期性、可扩展性和工作产物质量， 但应通过明确事件、仓储接口和应用服务与 Task Runtime 协作。

通用域

Profiles、Settings、Updater、Telemetry、Native Helpers、打包脚本等属于通用域。它们对产品可用性很重要，但不应主导核心模型命名。

建模风险：当前项目文件名中大量使用 Service/Repository，领域边界主要靠目录和文档表达。随着功能继续增长，建议把核心域对象的生命周期和规则进一步沉淀为领域层类型，而不是只在 Electron service 中隐式实现。

11. 后续建模建议

  1. 建立 Context Map 文档：将本报告中的 15 个限界上下文固化为架构文档，明确每个上下文的 upstream/downstream、事件和所有权。
  2. 强化聚合边界：Task、ManagedSession、Channel、Memory、HeartbeatRun 应明确哪些子对象只能通过聚合根修改，避免跨 Repository 任意写入。
  3. 抽出领域事件命名：将 TaskCreated、TaskCompleted、ArtifactCreated、ApprovalRequested、MemoryCandidateProposed 等事件命名固化，减少 payload 语义漂移。
  4. 为策略对象建测试矩阵：PermissionRule、TaskStrategySnapshot、ChannelSpecialization、HeartbeatPulseDecision、SkillApplication 是高价值策略对象，应有规则级测试。
  5. 建立统一对象词汇表：README、docs 和 UI 中的 Agent、Managed Agent、Role、Session、Task、Run 应统一命名，避免用户和开发者混淆。
